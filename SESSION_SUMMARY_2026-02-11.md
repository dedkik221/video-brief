## 9) Архитектура проекта (добавлено для следующего этапа)

Цель: разделить будущие услуги на независимые страницы, чтобы правки не ломали соседние формы.

Предлагаемая структура:

- `/index.html` — мастер-страница (навигация по услугам)
- `/services/video/index.html` — видеосъёмка
- `/services/live/index.html` — прямая трансляция
- `/services/photo/index.html` — фотосъёмка
- `/services/tech/index.html` — техническое обеспечение мероприятий
- `/assets/css/app.css` — общие стили
- `/assets/js/core.js` — общее ядро (валидация, UI-утилиты, экспорт)
- `/assets/js/service-*.js` — логика конкретной услуги

Почему одинаковые `index.html` не конфликтуют:

- `index.html` может быть в каждой папке, это норма для веб-структуры.
- Разница определяется путём URL:
  - `/` (мастер)
  - `/services/video/`
  - `/services/live/`
  - и т.д.

## 10) Jira-интеграция (упрощённый целевой вариант)

Цель: после заполнения формы создавать задачу в Jira и прикладывать PDF как вложение.

Целевой flow:

1. Пользователь заполняет форму услуги.
2. Нажимает "Отправить".
3. Backend принимает payload формы.
4. Backend создаёт issue в Jira (например `summary = "Заявка: Видеосъёмка"`).
5. Backend прикрепляет PDF с заполненным ТЗ в attachments задачи.

Принципиально важно:

- Прямой вызов Jira API из `index.html` не использовать (нельзя хранить Jira token в фронтенде).
- Интеграцию делать через серверный endpoint (например `POST /api/submit-brief`).

Минимальные данные для старта интеграции:

- `JIRA_BASE_URL`
- `PROJECT_KEY`
- `ISSUE_TYPE`
- сервисный Jira-аккаунт + API token
- маршрут, куда класть заявки внутри Jira-процесса
